# Master Prompt: Add QR Insertion Workflow For Documents Without Existing QR

You are building this feature inside the existing repo and must preserve the current QR replacement workflow exactly as it is.

## Feature goal

Add a new workflow that allows an authenticated owner to:

1. upload or use an existing document that does not already contain a QR code
2. open a separate new page dedicated to QR insertion
3. manually select the exact position where a new QR code should be inserted
4. generate a new app-hosted QR code using the same access logic already used by QR replacement
5. insert that QR code into the selected PDF position
6. store the processed PDF privately using the same secure file handling rules as the existing workflow
7. download the processed PDF
8. download the generated QR image alone as a PNG

This is an additive feature only.
Do not break, replace, or interfere with the existing QR replacement workflow.
The current “replace existing QR” path must keep working unchanged.

## Functional requirements

### 1. Separate workflow

- Add a new page and flow specifically for “insert new QR”.
- Do not overload the current “replace QR” flow in a way that makes the logic confusing or fragile.

### 2. Manual placement

- The owner must manually select the QR rectangle position.
- Reuse existing coordinate conversion and selector logic where possible.
- Persist exact PDF-space coordinates.

### 3. QR generation

- Use the same stable target URL logic already implemented in the repo.
- Support:
  - verify
  - open
  - download
- Keep raw storage URLs out of the QR payload.

### 4. PDF mutation

- Insert the generated QR into the chosen rectangle.
- Do not rebuild the whole PDF from images.
- Use `pdf-lib`.
- Preserve page size and all non-QR content.
- Since this is insertion into an empty area, do not first erase an old QR unless the chosen design explicitly requires a white background rectangle for readability.
- If a white background rectangle is used before drawing the QR, keep that behavior explicit and minimal.

### 5. Storage and access

- Store processed PDFs privately in Supabase Storage like the current workflow.
- Keep owner access app-mediated.
- Keep public access governed by the same verification and secure streaming logic.

### 6. QR image download

- Provide a safe owner-only way to download the generated QR image as PNG.
- Do not make QR image URLs public by default unless there is an explicit policy reason.

### 7. Management integration

- The inserted-QR documents should appear in owner document management in a way that does not confuse them with replaced-QR documents.
- If needed, add a clear document processing type such as:
  - `REPLACE_EXISTING_QR`
  - `INSERT_NEW_QR`
- Only add schema changes if they materially improve clarity and maintainability.
